Model Evaluation — Confusion Matrix & Metrics

W10 · Assignment Q1▼
Predicted +Predicted −
Actual +TPFN
Actual −FPTN
Accuracy
$$\text{Acc}=\frac{TP+TN}{TP+TN+FP+FN}$$
Precision
$$P=\frac{TP}{TP+FP}$$

"Of all predicted positive, how many actually are?"

Recall (Sensitivity)
$$R=\frac{TP}{TP+FN}$$

"Of all actual positive, how many did we catch?"

F1-Score
$$F1=\frac{2\cdot P\cdot R}{P+R}=\frac{2\cdot TP}{2\cdot TP+FP+FN}$$

Harmonic mean — penalises if either P or R is very low

Core Exam Trap (Assignment Q1)
Dataset: 10,000 transactions, 50 fraud. Model predicts all negative.
  • TP=0, FP=0, FN=50, TN=9950
  • Accuracy = 9950/10000 = 99.5% ← looks great, is useless
  • Recall = 0/50 = 0% ← catches zero frauds!
  • F1 = 0 ← the honest metric
Why Accuracy fails: when classes are severely imbalanced, always predicting majority = high accuracy but no learning. Use F1, Precision, Recall instead.
How to Answer "Construct Confusion Matrix"
1. Identify TP, FP, FN, TN from the scenario.
2. Draw the 2×2 table and label all cells.
3. Calculate requested metrics with formula.

Confusion matrix: [[90,10],[5,95]]. Compute Precision, Recall, F1.

Show Answer
TP=95, FP=10, FN=5, TN=90
Precision = 95/(95+10) = 0.905
Recall = 95/(95+5) = 0.950
F1 = 2(0.905)(0.950)/(0.905+0.950) = 0.927

SMOTE & Cost-Sensitive Learning

W10 · Assignment Q1.3▼

SMOTE — Synthetic Minority Over-sampling:

Geometric Mechanism (How to explain in exam)
For each minority sample $\mathbf{x}_i$, find its K nearest neighbours within the minority class. Select a random neighbour $\mathbf{x}_{nn}$. Generate a new synthetic sample by interpolating: $$\mathbf{x}_{new} = \mathbf{x}_i + \lambda(\mathbf{x}_{nn}-\mathbf{x}_i), \quad \lambda\in[0,1]$$ Creates points along the line segment between two minority neighbours — not random noise.

Advantage:

Increases minority class density without pure duplication. Better generalisation than simple oversampling.

Limitation:

Can create noisy synthetic samples in overlapping regions. Doesn't clean majority class.

Cost-Sensitive Learning:

Mechanism
Modify the loss function to penalise misclassifying minority class more heavily. E.g., weight fraud misclassification 100× more than non-fraud. No new data created — the model learns to care more about minority errors.

Advantage:

No data augmentation risk. Directly targets the optimisation objective.

Limitation:

Requires choosing good class weights (often domain-knowledge). Doesn't add information to the model.

Data Leakage — Most Tested Post-Mid Topic

W11 · Assignment Q3▼
Definition
Data leakage = including information about the target variable in training data that would NOT be available at prediction time.

Type 1: Train-Test Contamination

Fitting preprocessing (StandardScaler, TargetEncoder) on the full dataset before splitting. Test set statistics "leak" into the training pipeline.

Wrong Order (Assignment Q3)
❌ Fit scaler on all data → split → train model
✅ Split first → fit scaler on train only → apply to test

Correct Pipeline:

1
Split into train/test FIRST
2
Fit StandardScaler on train only
3
Transform (apply) scaler to both train and test
4
Fit TargetEncoder on train only
5
Transform test with the train-fitted encoder
6
Train model on transformed train set

Type 2: Target Leakage

Using features that are only available after the event you're predicting. E.g., using a patient's post-diagnosis treatment data to predict the diagnosis.

Assignment Q3.3 — Refuting "20% leakage is negligible"
The argument fails because: even 20% of test data having inflated (over-optimistic) evaluation metrics causes you to select worse models, tune hyperparameters incorrectly, and deploy overconfident systems. The leakage affects the validity of your evaluation — not just 20% of data. A model selected under leakage may perform significantly worse in production, regardless of leakage proportion.

TargetEncoder Leakage Trap:

If you compute city→mean(target) on the full dataset including test, test labels flow back into features. Must compute target encoding statistics on train data only.

Feature Encoding & Cyclical Features

EncodingHowProblem
Label EncodingCat→integer (0,1,2…)False ordinal: Karachi(2) > Lahore(1) is meaningless
One-Hot EncodingBinary column per categoryHigh dimensionality; 1000 cities → 1000 cols

Target Encoding Example
City=Lahore appears in rows with bought=[1,0] → mean=0.5
City=Karachi rows: bought=[0,0] → mean=0.0
City=Multan rows: bought=[1] → mean=1.0

Cyclical Features (hour, day, season):

The Problem
Hour 23 and hour 0 are 1 hour apart, but numerically $|23-0|=23$ — model sees them as far apart.
Cyclical Encoding
$$x_{\sin}=\sin\!\left(\frac{2\pi x}{T}\right), \quad x_{\cos}=\cos\!\left(\frac{2\pi x}{T}\right)$$

$T=$ period (T=24 for hours, T=7 for days of week)

Hour 23: $\sin(2\pi\cdot23/24)=-0.26$, $\cos=-0.97$. Hour 0: $\sin=0$, $\cos=1$. Distance in sin/cos space is small — correctly close!

Data Drift vs Concept Drift

W13–W14 · Assignment Q4.1▼

Data Drift (Covariate Shift)

$P(X)$ changes — input distribution shifts. But $P(Y|X)$ stays same — true relationship unchanged.

Ride-sharing example: Model trained on pre-COVID commuting patterns. After COVID, work-from-home changes who requests rides (population mix shifts). Same model logic, different data.

Detected by: feature statistics, KS test on input features.

Concept Drift

$P(Y|X)$ changes — the true mapping changes. Same inputs now produce different correct outputs.

Ride-sharing example: New ride-share competitors enter market, changing demand patterns for same time/location features. The true relationship between features and demand has changed.

Detected by: model accuracy metrics, label comparison.

AspectData DriftConcept Drift
What changes$P(X)$$P(Y|X)$
Inputs different?YesNot necessarily
Labels shift?NoYes
RiskMediumHigh — model fundamentally wrong
ActionMonitor, maybe retrainRetrain immediately

Types of Concept Drift:

  • Sudden: COVID disrupts demand overnight
  • Gradual: Customer tastes evolve over months
  • Incremental: Inflation slowly erodes price-based fraud thresholds
  • Recurring: Holiday shopping patterns repeat seasonally

KS Test — Kolmogorov–Smirnov

W13–W14 · Assignment Q4.2▼
KS Statistic
$$D=\sup_x|F_{train}(x)-F_{prod}(x)|$$

$F$ = empirical CDF | $D\in[0,1]$ | $D=0$ → identical distributions

D valueInterpretationAction
D < 0.1No drift — stableContinue monitoring
0.1 ≤ D < 0.2Mild driftMonitor closely
D ≥ 0.2Significant drift!Investigate, retrain
p < 0.05Statistically significantDrift confirmed
Assignment Q4.2 Answer Structure
(a) Compares: training feature distribution $F_{train}$ vs current production distribution $F_{prod}$.
(b) Measures: the maximum vertical gap between the two CDFs (supremum of absolute differences).
(c) Response: p<0.05 triggers alert → investigate root cause → schedule retraining or feature review.

Adversarial Attacks & Red Teaming

W15 · Assignment Q4.3▼

Attack Types:

Evasion Attack (inference time)

Modify input to fool the model. Stop sign + sticker → misclassified as speed limit.

Poisoning Attack (training time)

Inject malicious samples into training data to corrupt model behaviour.

Assignment Q4.3 — Adversarial in Tabular Data
Image domain: Add imperceptible pixel noise → model misclassifies (human can't see change).

Tabular regression (ride-sharing): Adversarial input = deliberately crafted feature values (e.g., manipulate hour=11pm as if it's 3am by encoding tricks, or inject extreme feature values) to cause the demand model to output incorrect predictions. No pixels — instead perturbing numerical/categorical feature values. Key difference: tabular adversarial perturbations must stay within plausible feature ranges (e.g., temperature can't be 500°C) to avoid detection — harder to craft but still possible.

Red Teaming (for regression model):

Method (for exam)
Before deployment, create a structured adversarial test suite: (1) craft boundary inputs (unusual feature combinations), (2) test with outlier feature values, (3) check for feature-level sensitivity (what happens if one feature is 10× its normal value?), (4) test under distribution shift scenarios (inputs from different time periods). Accept the model only if MAE stays within tolerance across all adversarial inputs.

LLM-Specific Attacks:

  • Prompt injection: "Ignore previous instructions..."
  • Jailbreaking: roleplay, Base64 encoding, translation tricks

Privacy Attacks:

  • Model inversion → reconstruct training data
  • Membership inference → was X in training set?
  • Model extraction → steal model via queries

Fairness Metrics

W15▼
Demographic Parity
$$P(\hat{Y}=1|A=0)=P(\hat{Y}=1|A=1)$$

Same positive prediction rate across groups

Equal Opportunity
$$\text{TPR}_0=\text{TPR}_1$$

Qualified candidates have equal chance of positive outcome

Equalized Odds
$$\text{TPR}_0=\text{TPR}_1 \;\text{ AND }\; \text{FPR}_0=\text{FPR}_1$$
Calibration
$$P(Y=1|\hat{Y}=p,A=a)=p\;\forall a$$

Confidence equally reliable for all groups

MetricWhat equals across groups
Demographic ParityPositive prediction rate $P(\hat{Y}=1)$
Equal OpportunityTPR (true positive rate)
Equalized OddsTPR + FPR both equal
CalibrationConfidence reliability
Key Point
Fairness metrics can CONFLICT with each other and with accuracy. "Fair" is context-dependent — no single definition is always correct.
Quick Check
Group A: 560/800 approved (70%). Group B: 80/200 approved (40%).
Demographic Parity violated: 70% ≠ 40%.
